# 财务结算系统 — 项目介绍

> 以 RPA（机器人流程自动化）视角审视：将原先需要人工在 Excel 中逐行核对、计算、调配的财务结算工作流，抽象为一套可维护、可追溯的自动化系统。

---

## 一、项目定位

本系统服务于游戏/App 分发行业的收入结算场景，处理**渠道方、研发商、我方公司**三方的月度收入分成计算。

**RPA 思维映射：**

| 人工操作 | 系统替代 |
|---------|---------|
| 登录各渠道后台下载对账单 | 模板导入 / 弹性导入 / QuickSDK 对接 |
| Excel 中做 VLOOKUP 匹配游戏、渠道 | FK 解析器 + 游戏名称模糊匹配引擎 |
| 手工计算折扣、分成、扣款 | `settlement_formula.py` 单一公式权威 |
| 人工复核差异行 | 弹性导入差异对比视图 |
| 锁定已核实的金额 | 锁定系统（冻结终值，防止覆盖） |
| 月末按主体汇总应收应付 | AR/AP 快照 + 双分录账本 |

## 三、功能模块

### 3.1 基础数据管理

统一管理结算流程涉及的所有主数据实体。

| 实体 | 说明 |
|------|------|
| 游戏信息 | 游戏编号、名称、折扣率 |
| 我方公司 | 我方签约主体公司，通过**项目编码**与游戏关联 |
| 研发商户 | 游戏研发商及其与游戏的映射关系 |
| 渠道管理 | 三级渠道层级：渠道类别 → 后台渠道 → 子渠道 |
| 分成配置 | 收入方/付款方的分成比例、渠道费率、税率，支持时间段生效 |
| 签约主体信息 | 法人信息（银行账号、税号），用于结算单全量导出 |

**关键设计：** 公司映射采用四级优先级解析链（渠道级精确匹配 → 游戏级兜底 → 项目编码回溯 → 渠道签约主体兜底），支持同一游戏在不同渠道归口到不同公司主体。

---

### 3.2 模板导入

标准批量数据录入通道。用户下载预设 Excel 模板，填写后上传，系统解析、校验、导入。

**流程：** 下载模板 → 填写数据 → 上传预览 → 逐行校验 → 确认导入

**支持的数据类型：** 游戏信息、渠道层级、分成配置、扣款数据、流水数据等十余种模板。

---

### 3.3 弹性导入

处理渠道方提供的**非标格式对账单**。

**核心能力：**
- **列映射推理：** 同义词词典 + 模糊匹配，自动推断上传文件列与系统字段的对应关系
- **游戏名称匹配：** 多 pass difflib 级联（名称相似度 → 前 3 候选 → 金额一致性校验 → 比率校验）
- **差异对比：** 展示系统已有值与导入值的差异，用户逐行审核
- **UPDATE-only 规则：** 仅修正已有扣款/配置数据，不创建新条目（符合 ADR-003）

**保护机制：** 导入前检查目标行是否被锁定，存在锁定值则拒绝导入。

---

### 3.4 OCR 导入（Beta）

截图对账单识别通道。基于 PaddleOCR 的中文表格识别能力。

**架构：** 主后端（FastAPI :8770）+ OCR 桥接服务（独立 venv，PaddleOCR :8771），10 分钟无活动自动关闭以释放约 1GB 显存/内存。

---

### 3.5 结算计算 & 查询

核心业务模块，展示渠道侧和研发侧的月度结算明细。

**计算引擎（`settlement_formula.py`）：**
```
净收入 = 原始流水 × 折扣率 - 总扣款
结算金额 = 净收入 × 分成率 × (1 - 渠道费率) × (1 - 税率)
```

**单条公式权威：** 结算、锁定、快照统一走 `compute()` 纯函数，确保计算一致性。

**交互功能：**
- AG Grid 内联编辑：修改扣款明细，实时重算
- 锁定/解锁：点击单元格锁定 `real_revenue` 或 `settlement_amount`，锁定的值不因重算而改变
- 客户端 CSV 导出：尊重当前列筛选排序
- 全量导出：以原始交易粒度导出含三级渠道层级的完整结算单

---

### 3.6 锁定系统

将核实的终值冻结，防止后续导入或重算覆盖。

**设计要点：**
- 锁定值存储在独立的 `channel_locks` / `publisher_locks` 表（非公式表）
- 解锁后自动从 `games` 表读取当前折扣率重新计算
- 锁定 `settlement_amount` 时自动创建双分录账本（Dr 应收 / Cr 收入）
- 弹性导入前置锁检查（ADR-004）

---

### 3.7 AR/AP 快照

月度关账用。从渠道锁定数据汇总生成当月的应收/应付记录。

**流程：** 手动触发快照 → 按 `(实体, 月份)` 聚合锁定金额 → 写入 `arap_records` → 月度关账将锁定数据归属到工作月份。

---

### 3.8 仪表盘

收入排行的可视化概览。

- 渠道/研发收入排行榜
- 月度收入趋势图（ECharts）
- 支持按公司/渠道/游戏维度下钻

---

### 3.9 备忘录系统

按渠道/研发关联备注与提醒。支持文件附件、提醒日期。结果推送至结算查询页面。

---

### 3.10 系统管理

| 功能 | 说明 |
|------|------|
| 认证 | 首次运行设密码，SHA256 令牌鉴权，5 次失败 IP 锁 30s |
| 备份/恢复 | AES-256-CBC 加密备份（自动密钥 / 口令模式），三种加密等级 |
| 版本更新 | HTTP 远程检测版本 + ZIP 补丁热更新 |
| 局域网共享 | 可选绑定 `0.0.0.0`，自签名 SSL 证书 |

## 四、架构要点

### 4.1 三模块解耦

```
[基础层]         [弹性导入]         [锁定系统]
  models.py        flexible_import    lock_service
  settlement_       template_import    snapshot_service
  service
       │                 │                  │
       └── settlement_formula.py (公式权威) ──┘
       └── ChannelSettlement (共享宽表) ──────┘
```

三个模块通过 `settlement_formula.py` 和 `ChannelSettlement` 宽表契约解耦，互不直接依赖。

### 4.2 宽表模式

`raw_settlements` 是唯一的流水数据聚合源。导入时解析渠道层级并预聚合，运行时**零 JOIN** 查询。`ChannelSettlement` 向各模块提供统一的 `(channel_id, game_id, month)` 三维键视图。

### 4..3 三维唯一键

所有渠道侧实体使用 `(channel_id, game_id, month)` 作为唯一键，研发侧使用 `(publisher_id, game_id, month)`。消除了以 `game_id + month` 二元素键在跨渠道场景下的歧义。

---

## 五、适用场景

- 有多个渠道分发商、多款游戏、多家主体公司的游戏发行/联运业务
- 需要从非标渠道对账单中提取结算数据的场景
- 需要对已核对数据进行锁定保护，防止后续变更的场景
- 月度需要按公司主体汇总应收/应付的关账流程
